# Security Policy

## Reporting A Vulnerability

Please do not open a public GitHub issue for suspected vulnerabilities.

Instead, report security issues privately to the project maintainers using one
of the following paths:

- the repository owner's security contact channel, if one has been shared with
  you directly
- a private GitHub security advisory, once the repository enables that workflow

If neither path is available, contact the repository owner directly before
publishing any details.

## What To Include

Please include as much of the following as you can:

- a description of the issue
- affected areas of the project
- reproduction steps or proof of concept
- expected impact
- any suggested mitigation

## What Not To Post Publicly

Do not publish the following in public issues, PRs, or discussions before the
maintainers have had a chance to review:

- unpatched vulnerabilities
- exploit details
- secrets or credentials
- attack paths that materially increase risk to users or operators

## Scope

This policy applies to security concerns involving:

- smart contracts
- off-chain services
- credential handling
- deployment or infrastructure configuration
- issue and PR workflows if they expose project security risk

## Response Expectations

The project is early-stage, so response times may vary. Maintainers will still
aim to:

- acknowledge receipt in a reasonable time
- assess severity and scope
- communicate next steps when possible

## Disclosure

Please allow maintainers a reasonable window to investigate and mitigate before
public disclosure.
